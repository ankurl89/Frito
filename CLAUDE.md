## Architecture

### What this is

Frito AI is a D2C brand-building platform. A user signs up, chats with an AI co-founder to create a **Brand DNA** object, then adds products from a Qikink print-on-demand catalog. AI generates SVG product designs. Orders automatically route to Qikink for fulfillment.

### Core data model

`BrandDNA` (in `src/lib/types.ts`) is the central object everything derives from:
- `palette` (ColorPalette) — drives logo, design, and listing generation
- `niche`, `target_audience`, `voice_tone`, `brand_values` — drive AI prompt context
- `price_tier` — drives suggested sell prices via `suggestSellPrice()` in `src/lib/qikink-catalog.ts`

All AI generation routes receive the full `BrandDNA` as context, not just a name or niche.

### AI pipeline

All LLM calls go through **OpenRouter** via `src/lib/openrouter.ts`, which wraps the OpenAI SDK with `baseURL: "https://openrouter.ai/api/v1"`. OpenRouter does NOT support image generation — `openrouter.images.generate()` will 404.

Models:
- `MODELS.fast` — `anthropic/claude-haiku-4-5` (listings, quick tasks)
- `MODELS.smart` — `anthropic/claude-sonnet-4-5` (brand creation, design generation)

Image/logo generation uses **SVG via Claude** (`MODELS.smart`), not an image API. The routes in `src/app/api/ai/logo/route.ts` and `src/app/api/ai/design/route.ts` prompt Claude to output raw SVG, extract it with a regex, and return a base64 data URI (`data:image/svg+xml;base64,...`).

The design route uses a **two-pass approach**:
1. Creative director pass — Claude decides *what* to design given full brand context
2. SVG execution pass — Claude builds the actual SVG using that concept

**IMPORTANT — the design route generates ARTWORK ONLY (a standalone graphic on transparent background), never a product mockup.** Product mockups are generated by the Mockup Engine (see below). This enforces manufacturing accuracy: the customer sees a real supplier template with artwork composited at the print area, not an AI illustration of the product.

### Product Visualization System (Digital Twin)

Per `CreatorStudio - Product Requirement.pdf`, every product is a **Digital Twin** with three layers:

1. **Layer 1 — Supplier Templates** (`src/lib/qikink-catalog.ts`): Real photos of blank products. Each `ProductTemplate` has `views.front/back/side`, each with a `print_area` `{ x, y, w, h }` in normalised coords (0–1) of the template image, plus `print_px_width/height` for the production file.

2. **Layer 2 — Design Placement Engine**: Implemented inside the Add Product flow (`src/app/dashboard/[brandId]/products/new/page.tsx`). Validates artwork (resolution ≥ 600×600), handles placement (scale, offsetX, offsetY), and rejects assets that can't be printed.

3. **Layer 3 — Mockup Generation Engine** (`src/components/MockupCanvas.tsx`): Client-side HTML Canvas component. Composites artwork onto the supplier template at the print area. The template image is **never modified** — only the artwork inside the print rectangle. Exposes `exportPNG()` (catalog mockup) and `exportProductionFile()` (high-res artwork at print DPI for fulfillment).

Mockups are saved as PNGs to the **`mockups` public Supabase Storage bucket** via `src/lib/storage.ts`. The product row stores:
- `artwork_url` — standalone artwork (SVG data URI)
- `mockup_url` — composited storefront image (PNG in storage)
- `production_file_url` — print-ready high-res PNG (PNG in storage)
- `placement` — `{ scale, offset_x, offset_y }` so mockups can be re-rendered

AI may be used later for lifestyle photography (background, model placement), but the **product itself in those images must come from the actual mockup** — AI cannot regenerate the product.

### Mission Control (internal ops cockpit)

Isolated internal operations platform at `/mission-control`. **Not** a separate app — a route group with separate staff auth + RBAC inside the same deployment (physical separation deferred until scale justifies it; Kafka/NestJS/K8s explicitly rejected as premature).

- **Staff isolation by authorization**: `staff_users` table (keyed to `auth.users`, has `role`). The MC layout calls `getStaff()` and redirects non-staff to `/dashboard`. Merchant sessions never see MC. All existing users seeded as `super_admin`; new signups are NOT staff.
- **RBAC** (`src/lib/mission-control/rbac.ts`): 6 roles (`super_admin/ops_manager/support_agent/finance_manager/growth_manager/read_only`) → explicit permission sets, deny-by-default. `hasPermission()` enforced at the layout (nav visibility) AND per action API (before the privileged write).
- **Immutable audit** (`admin_audit_log` + `src/lib/mission-control/audit.ts`): every privileged action logs actor/role/action/entity/before/after/ip. RLS deny-all except service client; no update/delete.
- **Staff data reads use the service client** AFTER `getStaff()` verifies the role — cross-tenant by design.

**Modules built:**
- **Executive** (`/mission-control`): GMV, profit, AOV, merchants, orders-by-state, fulfillment success rate, live queue/DLQ health — all from `getExecutiveMetrics()`.
- **Order Operations** (`/mission-control/orders` + `/[id]`): cross-merchant order list with state filter chips + search; drill-in shows full `order_events` timeline, provider payload, job history, financials. Actions (retry/cancel/refund) route through the **state machine + queue** and audit — MC never mutates `orders.status` directly.
- **Fulfillment Ops** (`/mission-control/fulfillment`): per-provider health (sandbox/live, success rate, capabilities), queue throughput, and the **dead-letter queue** with a permission-gated "revive" action (`/api/mission-control/jobs/[id]/revive`).

**Action APIs** under `/api/mission-control/*` all follow: `requirePermission(perm)` → mutate via engine/state-machine → `logAdminAction()`. Staff entry point is a role-gated "Mission Control" link in the merchant dashboard sidebar (`isStaff` prop).

**Deferred MC modules** (nav stubs marked SOON): Merchants 360, Customers 360, Pricing, Inventory, Risk, Audit browser, AI Ops Copilot.

### Fulfillment Core (provider-agnostic order pipeline)

**The platform never talks to a fulfillment provider directly** — only through adapters. See `docs/FULFILLMENT.md` for the full design.

- **Canonical order states** (`src/lib/orders/states.ts`): `pending_payment → paid → fulfillment_pending → submitted_to_provider → in_production → packed → shipped → delivered`, plus `cancelled/failed/refunded`. Legacy values were migrated. This is the single source of truth — `ORDER_STATUS_META` (badges) and `customerStageIndex` (storefront tracker) both derive from it.
- **State machine** (`src/lib/orders/state-machine.ts`): `transition()` is the ONLY writer of `orders.status`. Validates legality, uses optimistic concurrency (`.eq("status", from)`), writes an append-only `order_events` audit row. Out-of-order/duplicate provider webhooks are idempotent (same-state no-op, illegal skipped not thrown).
- **Provider adapters** (`src/lib/fulfillment/adapters/`): implement `FulfillmentProvider` (`src/lib/fulfillment/types.ts`). `QikinkAdapter` is the real API + a sandbox mode (no creds → simulated submission). `PrintroveAdapter` is a stub proving the seam. `registry.ts` selects by `products.fulfillment_provider`.
- **Fulfillment Engine** (`src/lib/fulfillment/engine.ts`): the only caller of adapters. `submitOrder` is idempotent (skips if `provider_order_id` exists). Sandbox orders self-progress via an `order.simulate_advance` job chain; real orders progress via webhooks.
- **Durable queue** (`src/lib/queue/job-queue.ts` + `claim_job` SQL fn): Postgres outbox. `claim_job()` uses `FOR UPDATE SKIP LOCKED`. Backoff 1m→5m→15m→1h→24h then dead-letter. NOT a new vendor — swappable behind the small `enqueue/claimNext` surface.
- **Worker** (`/api/jobs/worker`, secret-gated): drains up to 10 jobs/call. Driven by `kickWorker()` inline after enqueue + pg_cron in prod. Dev uses `DevWorkerTicker` (localhost-only) → `/api/jobs/tick`.

**Service-role client** (`src/lib/supabase/service.ts`): workers/webhooks/state-machine have no user session, so they use this RLS-bypassing client. Typed as broad `SupabaseClient`. Never import into user-facing routes without authz.

**Checkout** (`/api/store/orders`): idempotent via `idempotency-key` header. Creates order as `paid`, transitions to `fulfillment_pending`, enqueues `fulfillment.submit`, kicks worker. Prices are re-derived server-side (never trusts client price). Never submits to a provider synchronously.

**Webhook** (`/api/webhooks/qikink`): now provider-structured — verifies signature via adapter, normalizes to canonical event via adapter, resolves order via `fulfillment_orders.provider_order_id`, applies transition. Adding/swapping a provider needs no change here.

### Customer-Facing Storefront

Every brand gets a public, customer-facing store at `/store/[slug]` — no auth required, RLS-policied for public read of brands/products/orders.

**The storefront is theme-derived from the Brand Book.** No template picker. CSS custom properties are set on the storefront root from the brand's palette + typography:
```
--brand-primary, --brand-secondary, --brand-accent, --brand-bg, --brand-text
--brand-headline-font, --brand-body-font
```
Every storefront component uses these vars exclusively for colors/fonts — so each brand feels distinct without any per-brand code.

**Subdomains for production.** Current routing is path-based (`/store/otakudistrict`). For production with wildcard DNS (`*.tryfrito.com`), `proxy.ts` should resolve subdomain → slug and internally rewrite to `/store/[slug]`. The DB has `brands.slug` (unique) ready for this.

**Page tree:**
- `/store/[slug]` — Homepage: hero, featured products, brand story, category grid, brand CTA
- `/store/[slug]/p/[productId]` — Product detail: mockup gallery, size selector, qty, add-to-cart, related products
- `/store/[slug]/collection/[category]` — Collection page (`all` shows everything)
- `/store/[slug]/cart` — Cart (zustand store persisted in localStorage, scoped by `brand_slug`)
- `/store/[slug]/checkout` — Address → Payment (Razorpay stub) → calls `/api/store/orders`
- `/store/[slug]/order/[orderId]` — Customer order tracking with 5-stage pipeline (Received → In Production → Packed → Shipped → Delivered). **Never exposes Qikink.**

**Customer order API: `POST /api/store/orders`** — public endpoint, no auth. Validates brand by slug, creates one order row per line item with computed cost/profit, awards `order_received` XP + checks revenue milestones for the brand owner (so the founder gamification fires from customer purchases).

**Brand slugs** are auto-generated in `POST /api/brands` (kebab-case from name, conflict-resolved with `-2`/`-3` suffix). Existing rows were backfilled.

**RLS additions:**
- `brands` — public SELECT where `slug IS NOT NULL`
- `products` — public SELECT where `status IN ('published', 'active')`
- `orders` — public INSERT (anyone can create); public SELECT (for order tracking page)

The owner-facing dashboard has a **"View store →"** link on the brand overview that opens the storefront in a new tab.

### Founder Progression (Gamification)

The product is framed as a founder game — every meaningful action awards XP, contributes to a streak, and may unlock achievements.

**Constants live in `src/lib/founder-constants.ts`** (zero server deps, safe to import client-side):
- `LEVELS` — 10 levels (Dreamer → CEO) with `min_xp` thresholds
- `XP_RULES` — action → XP map (e.g. `brand_created: 100`, `order_received: 100`, `revenue_1l: 1000`)
- `ACHIEVEMENTS` — badge catalog with icons, descriptions, categories
- `LAUNCH_STEPS` — the 5-step launch checklist
- `REVENUE_MILESTONES` — INR thresholds (₹1k → ₹10L)
- Helpers: `levelForXP()`, `nextLevel()`

**Server engine in `src/lib/founder-engine.ts`** (server-only):
- `awardXP(userId, action, metadata)` — logs the earn, increments `profiles.xp`, recalculates `profiles.level`, updates `current_streak` / `longest_streak`, unlocks any triggered achievements. Returns whatever was newly unlocked.
- `unlockAchievement(userId, key, metadata)` — direct one-time unlocks (idempotent via unique constraint on `(user_id, achievement_key)`).
- `checkRevenueMilestones(userId, brandId)` — sums total revenue across all of the user's brands and awards any newly-crossed milestone XP.

**Triggers wired into existing flows:**
- `POST /api/brands` → `brand_created` (Brand Builder badge)
- `POST /api/products` → `product_created` (Product Creator badge) + `product_published` if launched immediately
- `PATCH /api/products/[id]` → `product_published` (Launch Day badge) when status transitions
- `POST /api/ai/brand-book` → `brand_book_ready` on completion
- `POST /api/webhooks/qikink` → `order_received` (First Sale trophy) on first transition to confirmed + revenue milestone check
- `POST /api/founder/award-signup` → `account_created` (Dreamer badge) called from signup page

**Read APIs:**
- `GET /api/founder/me` — full snapshot: profile (xp/level/streak), current+next level + progress %, achievements, recent XP log, launch checklist with per-step completion
- `GET /api/founder/first-order` — first order details for the celebration overlay

**UI components:**
- `<FounderStatus />` — dark level bar + streak chip + launch checklist + achievement grid. Polls `/api/founder/me` every 8s and toasts newly-unlocked achievements (compares against `seenAchievements` ref).
- `<FirstSaleCelebration />` — full-screen confetti overlay with LinkedIn share. Shown once; dismissal persisted in localStorage (`first_sale_celebrated=1`).
- `AchievementToast` (internal to FounderStatus) — purple/yellow gradient badge with icon

Both components are mounted at the top of `/dashboard/[brandId]/page.tsx` so they're visible whenever the user opens the overview.

### AI Brand Strategist & Brand Book

Every brand has two layers of identity:

1. **Brand DNA** — lightweight (name, tagline, story, palette, voice, audience). Generated conversationally during onboarding. Stored directly on the `brands` row.

2. **Brand Book** — the full brand operating system. 17 sections (foundation → strategy → persona → personality → archetype → tone → messaging → visual identity → guidelines → product positioning → pricing → competition → packaging → social → email). Stored in `brands.brand_book` JSONB. Generated by `POST /api/ai/brand-book` after the user launches their brand.

Status field `brands.brand_book_status`: `pending` → `generating` → `ready` (or `failed`).

The viewer page is `/dashboard/[brandId]/brand` — a long agency-style document with chapters. It polls every 3s while generation is in progress.

**Brand Memory System.** Downstream AI routes (currently `api/ai/listing`) read `brand.brand_book` and inject a memory block: archetype, "we are / we are not" voice, personality traits, sample headlines, and copy examples. This is what the PRD calls the "Brand Consistency Engine" — every generated asset reflects the documented brand.

When adding new AI-driven content (emails, social posts, ad copy), follow the same pattern: read `brand_book` from the brand record, pass relevant sections into the prompt as a memory block.

### Product Visualization Engine (PVE)

Server-side, Sharp-based rendering pipeline that produces real high-res storefront assets from the Digital Twin — replacing the old browser-canvas compositing. Core principle unchanged: **never AI-generate the product image**; composite validated artwork onto a real supplier template.

- **Rendering** (`src/lib/pve/renderer.ts`): Sharp pipeline at 2048px. Contains the template on a neutral canvas, fits artwork into the print area with **fabric-aware blending** (extracts the garment region under the print, soft-lights it into the artwork so wrinkles/shadows read through). SVG artwork rasterized at 300 density. Produces 4 assets: `primary` (2048², hero), `closeup` (print-area crop, 1024²), `thumbnail` (512²), `production_file` (artwork at `print_px_width/height`, transparent, no template — the printer file). All uploaded to the public `product-assets` Supabase bucket.
- **Validation** (`src/lib/pve/validation.ts`): resolution, effective DPI at production size, transparency (apparel), aspect-ratio fit, print-area coverage. Returns `error`/`warn` issues.
- **Per-product Visualization Profiles** (`src/lib/pve/profiles.ts`): threshold AND factor weights AND hard requirements differ per product. Resolved by category with per-SKU overrides. e.g. Apparel needs 85 + transparency; Poster needs 80 + 220 DPI + no transparency; Mug needs 80, brand-weighted; Phone case (override) needs 55%+ coverage, no transparency. Validation + scoring both take the profile.
- **Scoring** (`src/lib/pve/score.ts`): 0–100 across the same 5 factors (Resolution & DPI, Print Readiness, Placement Coverage, Brand Consistency, Aspect & Quality) but weighted per the product's profile; `publishable = total >= profile.threshold`.
- **API** (`/api/pve/render`, `runtime = "nodejs"`): validate → render → score → persist. Stores `product_assets` rows + `products.visualization_score` + `products.validation_report`, and sets `mockup_url`/`production_file_url` from the rendered assets.
- **Asset store**: `product_assets` table (Digital Twin) with a broad `asset_type` taxonomy — slots for `back/side/angle_45/folded/fabric_detail/lifestyle/3D` exist but aren't generated yet (honest call: don't fabricate angles from a single front template; populate when real multi-angle supplier templates land).

**Wired into:** Add Product (creates draft → renders → publishes if score ≥ 85, else keeps draft), Replace Artwork (re-runs PVE, respects order-protection), product edit page (asset gallery + score breakdown), and the storefront product gallery (uses `primary`/`closeup` assets).

Brand-color match in scoring uses Sharp `.stats().dominant` vs the brand palette.

**Artwork generation is a replaceable provider** (`src/lib/images/`): `ImageProvider` interface, `registry.ts` picks the highest-quality configured provider.
- `FluxFalProvider` (`FAL_KEY` set): real raster artwork via fal.ai Flux → **background removal (birefnet) → transparent PNG**. Two-step because Flux generates on a background but print artwork needs transparency.
- `SvgClaudeProvider` (default, no key): Claude emits SVG. Lower ceiling, zero dependency.

`/api/ai/design` builds the creative direction prompt, calls `getImageProvider().generateArtwork()`, and **re-hosts remote raster output to the `product-assets/_artwork/` bucket** (stable URL, CORS-safe for the browser placement preview). The PVE renderer consumes either a PNG URL or an SVG data URI transparently.

### Product Lifecycle & Management

Every product is a long-lived entity with full lifecycle management (per the Merchant Product Management PRD).

**Lifecycle states:**
- `draft` — created but not visible on storefront
- `published` — visible to customers
- `archived` — removed from storefront, retained internally with all history
- *Deleted* — only when product has zero orders. Products with orders are **auto-archived instead of deleted** (order protection).

The legacy value `active` is treated as equivalent to `published` everywhere (`STATUS_META` and product list queries normalise it).

**Order protection** — implemented in `src/app/api/products/[id]/route.ts`:
- If a product has any order, PATCHing `artwork_url` or `production_file_url` returns 409.
- DELETE on a product with orders becomes auto-archive (returns `{ action: "archived", message: ... }`).
- Merchant must duplicate the product to make design changes after fulfillment has begun.

**Lifecycle helpers** are in `src/lib/product-lifecycle.ts`:
- `snapshotVersion()` — writes a row to `product_versions` and bumps `products.version`. Every PATCH and POST records one.
- `logActivity()` — appends to `product_audit_log`. Every action (create/publish/unpublish/archive/duplicate/artwork/pricing/edit) gets logged.
- `hasActiveOrders()` — used to gate destructive actions.
- `STATUS_META` — shared label + Tailwind class lookup so badges look identical across pages.

**API surface:**
- `GET /api/products?brand_id=&status=&q=&sort=` — list with status filter, search, sort
- `POST /api/products` — create (snapshots v1, logs `create`)
- `GET /api/products/[id]`
- `PATCH /api/products/[id]` — edit (auto-detects change type: edit/artwork/pricing/publish/unpublish/archive, snapshots + logs)
- `DELETE /api/products/[id]` — hard delete OR auto-archive if has orders
- `POST /api/products/[id]/duplicate` — clones as draft with auto-suffixed name (V2, V3…)
- `GET /api/products/[id]/versions` — version history (newest first)
- `GET /api/products/[id]/audit` — activity log (last 50)

**Pages:**
- `/dashboard/[brandId]/products` — list with status tabs (All/Draft/Published/Archived), search, sort, per-card action menu (publish/unpublish/archive/duplicate/delete)
- `/dashboard/[brandId]/products/[productId]` — edit page with all fields + sticky action header + right-side Activity/Versions tabs
- `/dashboard/[brandId]/products/[productId]/artwork` — replace artwork flow (reuses `MockupCanvas`, regenerates mockup + production file on save)

### Route protection

`src/proxy.ts` (Next.js 16's replacement for `middleware.ts`) guards routes:
- `/dashboard/*` and `/onboarding` → redirect to `/login` if unauthenticated
- `/login` and `/signup` → redirect to `/dashboard` if already authenticated
- `/api/webhooks/*` is excluded from the matcher (public webhook endpoint)

### Supabase setup

Two clients: `src/lib/supabase/client.ts` (browser, `createBrowserClient`) and `src/lib/supabase/server.ts` (server components/routes, `createServerClient` with cookies). Always use the server client in API routes and Server Components.

Schema is in `supabase-schema.sql`. Tables: `brands`, `products`, `orders`, `profiles`. All tables have Row Level Security — users can only access their own data. The `handle_new_user` trigger auto-creates a `profiles` row on signup.

### Product catalog

`src/lib/qikink-catalog.ts` contains a static array of 10 Qikink products with base prices in INR. This is a curated mock — in production it should be fetched live from Qikink's API. Products have `qikink_product_id` (e.g. `"QK-001"`) that maps to Qikink's catalog.

`suggestSellPrice(baseCost, priceTier)` applies a multiplier (2.0x–4.5x depending on tier) and rounds to the nearest ₹50.

### SKU generation

`src/app/api/products/route.ts` generates SKUs on product creation using the first 4 chars of `brand_id` + a 6-char random string from `src/lib/nanoid.ts`.

### Qikink fulfillment webhook

`src/app/api/webhooks/qikink/route.ts` receives POST webhooks from Qikink when order status changes. Register the public URL `https://yourdomain.com/api/webhooks/qikink` in the Qikink dashboard. The webhook maps Qikink status strings to internal `OrderStatus` enum values and updates the `orders` table.

## Key non-obvious decisions

- **No Shopify for MVP.** The platform hosts its own storefront. Shopify export is a future premium feature.
- **No n8n.** Background jobs that would normally go through workflow automation are handled directly in API routes or via Inngest (not yet wired up — currently synchronous).
- **No Fabric.js.** Design editing is AI-only (describe → generate → approve). There is no canvas editor.
- **Email confirmation is disabled** in Supabase (`mailer_autoconfirm: true`) so signups work immediately without email verification.
- **Supabase project:** `pecaekbgmbuhssrwtclx` (region: `ap-south-1`, Mumbai).
